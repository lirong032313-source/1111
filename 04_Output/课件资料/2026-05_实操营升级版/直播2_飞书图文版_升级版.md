# 🧠 认知篇：为什么你的知识库还是「死」的

---

## 一、灵魂拷问：你有知识库，但你用过吗？

**先回答 3 个问题：**

1. 上周你发的朋友圈感悟，现在在哪个文件夹？
2. 你脑子里刚冒出来的一个选题灵感，3 天后你还能找到吗？
3. 那个学员问你的好问题，你除了回复她，有没有存下来？

> **中 2 条以上？那你的知识库还只是「资料柜」，不是「会呼吸的大脑」。**

---

### 为什么「入库」这一步总是断掉？

你现在的状态：

```
灵感来了（微信/朋友圈/脑子里）
    ↓
要不要存进知识库？
    ↓
打开电脑 → 找到 Obsidian → 新建文件 → 写内容 → 分类归档
    ↓
发现写了 5 分钟还在想标题格式
    ↓
算了，下次再说吧 ❌
```

**核心矛盾**：你有了知识库（直播1搭好的三层架构），但很多同学入库的方式还是「手动搬砖」。入库摩擦力太大 = 知识库永远是空的。

> **这节课要解决的就是这件事——让你发一条微信，AI 自动帮你搬好砖。**

## 二、三个工具，三个角色｜一张图讲清楚

**直播 2 最核心的认知**：你即将用到的三个工具，各司其职。

```
        ┌─────────────────────────┐
        │   你（微信里发消息）      │
        └──────────┬──────────────┘
                   ↓
        ┌──────────────────────────────┐
        │  🦞 微信 ClawBot              │
        │  （信使 = 送消息的）          │
        │  你的微信消息 → 转给后台       │
        │  后台的回应 → 送回你的微信     │
        └──────────┬───────────────────┘
                   ↓
        ┌──────────────────────────────┐
        │  🐾 WorkBuddy                 │
        │  （助理大脑 = 处理任务的）     │
        │  ─ 理解你说的是什么意思        │
        │  ─ 决定调用哪个能力           │
        │  ─ 调度工作流执行              │
        └──────────┬───────────────────┘
                   ↓
        ┌──────────────────────────────┐
        │  📚 Obsidian + ob-大脑        │
        │  （知识库 = 你的文件仓库）     │
        │  ─ raw/ 你丢的原始素材         │
        │  ─ wiki/ AI 整理的信息卡       │
        │  ─ CLAUDE.md 规则书            │
        └──────────────────────────────┘
```

> **金句：「Obsidian 是大脑（装知识），WorkBuddy 是助理（处理任务），ClawBot 是信使（送消息）——三个工具各司其职，不是相互替代。」**

---

### 分工对比：谁干什么，不干什么

| 工具 | 干什么 | 不干什么 |
|---|---|---|
| **Obsidian** | 看/编辑/搜索本地 .md 文件 | 不发微信、不调 AI、不跑工作流 |
| **WorkBuddy** | 跑工作流、调度能力、读写文件、调 AI | 不负责消息收发 |
| **微信 ClawBot** | 把你的微信消息和后台连起来 | 不做任何 AI 计算 |

---

## 三、为什么分三层？—— 工程上的「关注点分离」

> 「你可能问：为什么不能一个工具搞定？我反过来问你——你公司的 IT 部门会让你电脑上同时跑『打印机+邮件+文件柜+ChatGPT』在同一个软件里吗？不会。**每个工具做自己最擅长的事，组合起来才稳定。**」

### 打通前 vs 打通后

```
打通前（直播1之后你的状态）：
  微信(碎片) ←→ Obsidian(孤岛) ←→ AI(陌生人)
  三者各干各的，你手动搬运，累且容易断

打通后（这节课之后）：
  微信 ←→ ClawBot(信使) ←→ WorkBuddy(助理) ←→ Obsidian(底座)
  你只负责「发微信」和「确认」，中间全自动
```

## 五、CLAUDE.md 和其他 4 个文件 —— 合同 vs 数字员工设定

**从刚才的旅程里你看到了——龙虾干活时同时读了两个文件：CLAUDE.md 和 MEMORY.md。它们干的不是同一件事。**

### 一张图说清：CLAUDE.md 是合同，4 个文件是设定

```
┌─────────────────────────────────────────────────────┐
│                                                     │
│   📝 CLAUDE.md                                      │
│   （合同 —— 直播1已写）                             │

│   「我雇佣你，工作内容是：                             │

│   → 这是你和 AI 之间的「雇佣合同」                     │
│     规定了工作职责、质量标准、边界条款                  │
│                                                     │
├─────────────────────────────────────────────────────┤
│                                                     │
│   🦞 4 个 WorkBuddy 文件                            │
│   （数字员工设定 —— 本节课配）                       │
│                                                     │
│   MEMORY.md → 工位在哪？（知识库路径）                │
│   SOUL.md   → 什么性格？（怎么跟你相处）              │

│                                                     │
│   → 这是这个「数字员工」入职时要填的员工档案           │
│     合同管「做什么」，设定管「怎么做、跟谁做」          │
│                                                     │
└─────────────────────────────────────────────────────┘
```

### 为什么合同和设定要分开？

| 类比 | 现实世界 | 你的 AI 系统 |
|---|---|---|
| **合同** | 劳动合同（职责/薪酬/解约条款） | **CLAUDE.md**（入库规则/写作风格/边界红线） |
| **员工档案** | HR 系统里的个人信息 | **MEMORY.md**（路径/项目/经验） |
| **岗位说明书** | 这个岗位的行为准则 | **SOUL.md**（性格/沟通方式/工作流程） |
| **老板偏好** | 你知道老板喜欢邮件还是当面汇报 | **USER.md**（你的习惯/讨厌什么/红线） |
| **工牌** | 胸前的名字和照片 | **IDENTITY.md**（名字/emoji/标签） |

### 一句话记住

> **「CLAUDE.md = 劳动合同，管龙虾『做什么』。4 个配置文件 = 数字员工入职档案，管龙虾『怎么做、跟谁做、在哪做』。合同 + 档案都齐了，这个数字员工才算真正入职。」**

