# SOP：Raw 素材 → Wiki 精华 → 公众号文章

> **类型**：工作流程 SOP
> **位置**：05_System/工作流程SOP/
> **覆盖场景**：外部学习资料 / 课程逐字稿 / 咨询记录 → 提炼进 Wiki → 产出公众号文章
> **执行者**：石榴 🐾（AI 搭档）
> **更新时间**：2026-05-24

---

## 一、工作流总览

```
01_Raw（原始素材）
       ↓  ① 读取 + 理解
02_Wiki（结构化精华）
       ↓  ② 基于 Wiki 选题 + 写作
04_Output/自媒体内容/公众号/草稿箱（公众号初稿）
```

**核心原则**（来自 Karpathy 理论）：
> Raw = 事实（只存不改）
> Wiki = 结构（AI 提炼后的知识）
> Output = 应用（基于 Wiki 生成内容）

**不要做的事**：
> ❌ 不要跳过 Wiki，直接从 Raw 写公众号
> 结果就是"正确的废话"，没有蓉蓉子的声音

---

## 二、Step 1：读取 Raw 原始文件

### 触发条件

| 触发方式 | 示例 |
|---|---|
| 用户主动说 "帮我整理 Raw 里的 XX 文件" | "把 Raw 里刚上传的 PDF 整理到 Wiki" |
| 自动化定期扫描（可选） | 每日 21:00 扫描 `01_Raw/` 新文件 |
| 用户上传文件到 Raw 后说 "处理这个" | "刚传了一个逐字稿，帮我整理" |

### 执行步骤

```
① 确认 Raw 里的目标文件
   - 优先处理今天/最近新增的文件
   - 检查 `01_Raw/已处理清单.md`，跳过已处理的

② 读取文件内容
   - .md 文件：直接 Read 读取
   - .pdf 文件：用 pymupdf (fitz) 提取文字层
   - .txt 文件：直接 Read 读取
   - 纯图片/扫描件：告知用户"该文件为图片，无法提取文字，建议提供文字版"

③ 判断文件类型 → 决定提炼方向
   ├─ 课程逐字稿   → 提炼方法论 → 6号房「我怎么想」
   ├─ 对标人物资料 → 建人物档案   → 案例库/对标人物_XX.md
   ├─ 咨询记录     → 提炼痛点/案例 → 5号房「我面对谁」+ 案例库
   └─ 学员对话     → 提炼需求/反馈 → 5号房学员档案
```

### 完成后

- 在 `01_Raw/已处理清单.md` 追加一行记录
- 格式：`| YYYY-MM-DD | 文件名 | ✅ 已提炼精华 | 提炼到 Wiki 路径 |`

## 三、Step 2：提炼精华到 Wiki

### 2.1 判断应该存到哪个房间

| 内容类型 | 存入路径 |
|---|---|
| 思考框架 / 方法论 / 价值观 | `02_Wiki/6号房-我怎么想/` |
| 人物档案 / 对标案例 | `02_Wiki/案例库/对标人物_XX.md` |
| 学员痛点 / 用户画像 | `02_Wiki/5号房-我面对谁/学员档案/` |
| 产品方法论 | `02_Wiki/4号房-我卖什么/` |
| 金句 / 观点 | `02_Wiki/3号房-我说过什么/金句库/` |
| 能力/工具总结 | `02_Wiki/2号房-我会什么/信息卡/` |

### 2.2 Wiki 文件格式规范

```markdown
# 📌 标题（格式：日期_主题.md）

> **来源**：原始文件名
> **入库时间**：YYYY-MM-DD
> **关联**：[[关联的其他 Wiki 文件]]

---

## 一、核心洞见（3-5 条）

（用蓉蓉子的语气，不是百度百科）

## 二、可行动点

- [ ] 可以直接落地的行动
- [ ] 需要蓉蓉子确认的方向

## 三、与蓉蓉子体系的关联

（说明这个方法论如何与现有六间房体系结合）

---

*本文件由 石榴🐾 于 YYYY-MM-DD 从 Raw/XX 提炼生成*
```

### 2.3 提炼质量要求

```
✅ 有蓉蓉子的声音（口语化、有温度）
✅ 有具体可行动点（不是泛泛而谈）
✅ 与现有 Wiki 建立 [[双向链接]]
✅ 标注"待蓉蓉子确认"的不确定信息

❌ 不要写成百科词条
❌ 不要遗漏原文的关键数据/案例
❌ 不要改变原文观点（只做结构化）
```

## 四、Step 3：基于 Wiki 写成公众号文章

### 3.1 选题决策

```
读取刚提炼的 Wiki 文件
        ↓
判断是否有公众号价值：
├─ 有痛点共鸣？   → 可以做 M1 成长日记类
├─ 有方法论干货？ → 可以做 M2 小而美产品类
├─ 有知识库/AI 技巧？→ 可以做 M3 智能体/知识库类
└─ 都不突出？     → 暂不写，等更好的角度

如果用户没有指定选题：
  生成 3 个标题选项 → 让用户选
```

### 3.2 调用 Wiki 顺序（写作前必读）

```
① 02_Wiki/6号房-我怎么想/（思考框架 + 价值观约束）
② 刚提炼的 Wiki 文件（核心内容来源）
③ 02_Wiki/3号房-我说过什么/金句库/（插入金句）
④ 02_Wiki/5号房-我面对谁/（用户痛点校准）
```

### 3.3 公众号文章格式规范

遵循 `2026-05-05_SOP_内容生产全流程.md` 中的规范：

**标题**：15-20 字，含痛点/反常识/数字
**结构**：
```
① 开头钩子（100-200字）
   - 故事切入 / 痛点共鸣 / 反常识
   
② 正文（3-4 个小标题，每个 300-500 字）
   - 场景 → 洞见 → 方法 → 行动
   
③ 结尾升华（100-200字）
   - 呼应开头，温暖收束
   - 行动召唤（关注/回复关键词）
```

**风格检查清单**：
```
✅ 第一人称"我"
✅ 温暖、口语化，像跟朋友聊天
✅ 长段落（不用列表代替正文）
✅ 有真实故事/场景
✅ 不说"正确的废话"

❌ 不用"不是...而是..."句式
❌ 不爹味说教
❌ 不编造案例
```

### 3.4 保存路径

```
04_Output/自媒体内容/公众号/草稿箱/
└── YYYY-MM-DD_文章标题.md
```

文件头信息：
```yaml
---
title: 文章标题
date: YYYY-MM-DD
status: 草稿
tags: [M1或M2或M3, 相关主题]
---
```

## 六、异常处理

| 异常情况 | 处理方式 |
|---|---|
| Raw 文件为图片/扫描件，无法提取文字 | 告知用户，请求提供文字版 |
| PDF 文字层不完整 | 用已有 .md 纪要补充，标注"部分内容来自纪要" |
| Wiki 中已有同类文件 | 不重复创建，改为在现有文件后追加章节 |
| 提炼后发现内容不适合写公众号 | 告知用户，暂不写，等更好角度 |
| 用户中途改变需求 | 停止当前流程，确认新需求后再继续 |
