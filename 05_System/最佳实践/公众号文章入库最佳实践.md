## 四、Claude Code 实战指南（核心）

> 这一节是这份 SOP 最重要的部分。**Claude Code 的强大不在于一次写对，而在于"试错→读错→改"的快速循环**。

### 4.1 第一次对话的开场套路

❌ **不好的开场**：
> "帮我下载公众号文章"

→ Claude Code 不知道你的脚本在哪、用什么方式、目标是什么。

✅ **好的开场**（参考 Step 3）：
- 指明文件位置（`download_wechat.py`）
- 给上下文（先读 README）
- 拆步骤（先试小量 → 再跑全量）
- 给授权（"出错你直接改"）—— 这一句关键

### 4.2 凭证怎么粘最安全

Claude Code 在你 Mac 本机跑，所以**直接粘到对话窗口是 OK 的**——它不会上传你的 cookie 到云端做训练。

但有几个习惯：
- 粘完之后**别在对话里反复重复 cookie 内容**（让 Claude Code 写到脚本里就好）
- 跑完任务后，可以让 Claude Code 把 cookie 字段清空（防止你哪天误把脚本发给别人）
- 如果你担心，让 Claude Code 把 cookie 写到 `.env` 文件而不是脚本里（再加 `.gitignore`）

### 4.3 出错时的高效协作

**这次你实际遇到的真实情况**：

第一次跑 `limit=5`，报错"未提取到任何文章"。Claude Code 干了什么：

1. **看返回的 JSON 结构** —— 它把 `appmsgpublish` 接口的真实响应保存下来分析
2. **发现字段名不对** —— 我之前写的是 `appmsgex` 和 `link`，但你公众号的接口实际返回 `appmsg_info` 和 `content_url`
3. **改了 `parse_publish_list` 函数** —— 把字段名换掉
4. **再跑** —— 154 篇全到位

> 💡 **这就是 Claude Code 的核心价值**：它不是一次性写对，而是**和真实环境快速对齐**。微信内部接口字段会变（甚至不同公众号略有差异），人工调试要查文档对字段，Claude Code 5 分钟搞定。

### 4.4 给 Claude Code 的协作三原则

| 原则 | 怎么做 |
|---|---|
| **给上下文** | 进对话先 `请先读 README.md` 或 `请先看 02_Wiki/...` |
| **给授权** | 主动说"出错你直接改"，不要让它每次都问你确认 |
| **给反馈** | 看到结果不对，**贴出实际输出**给它，比"它没跑"具体多了 |

### 4.5 实测有效的 Claude Code prompt 套路

**跑通后的延伸操作**（你也可以接着用）：


## 六、常见问题排错

| 报错 / 现象 | 可能原因 | Claude Code 怎么处理 |
|---|---|---|
| `cookie 失效 / 401` | 后台登录态过期（约 2-4 小时） | 让你重登后台、复制新 cookie 进来 |
| `freq control` | 触发风控 | 把 `delay_seconds` 调到 8，等 30 分钟再跑 |
| `未提取到文章` | 字段名变了 | 让它打印一次原始 JSON 响应，肉眼对字段，改 parser |
| 部分文章 `正文区域未识别` | 用了非常规模板（纯图、视频图文） | 跳过即可，最后报告会显示失败数 |
| 文件名乱码 | macOS 文件系统对特殊字符敏感 | `safe_filename` 函数已处理，应该不会出 |
