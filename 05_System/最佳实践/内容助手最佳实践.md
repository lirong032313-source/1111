## 一、内容助手 = 1 个 agent + 6 个 skill

```
┌─────────────────────────────────────────────────────┐
│        🦞 内容助手 = 内容助理 agent + 6 个 skill         │
├─────────────────────────────────────────────────────┤
│                                                     │
│   👤 内容助理（人格层）                               │
│      ↓ 提供写作风格、调用规则、边界                   │
│                                                     │
│   ⚙️  Skill 层（6 个原子能力）                         │
│   ┌──────────────┐ ┌──────────────┐                │
│   │ content-gen  │ │ wechat-ingest│ ← 入口路由         │
│   └──────────────┘ └──────────────┘                │
│   ┌──────────────┐ ┌──────────────┐                │
│   │topic-generator│ │ article-gen │ ← 创作主线         │
│   └──────────────┘ └──────────────┘                │
│   ┌──────────────┐ ┌──────────────┐                │
│   │xiaohongshu   │ │ daily-brief  │ ← 分发 + 监控     │
│   │  -gen        │ │              │                │
│   └──────────────┘ └──────────────┘                │
│                                                     │
│   📚 Wiki 底座（依赖）                                │
│   01-我是谁 / 02-我会什么 / 03-我说过什么 /            │
│   05-我面对谁 / 06-我怎么想 / CLAUDE.md             │
│                                                     │
└─────────────────────────────────────────────────────┘
```

**核心设计**：
- **1 个 agent**（内容助理）= 人格 + 调用规则 + 边界
- **6 个 skill** = 原子能力，单一职责，可独立调用
- **Wiki 底座** = 3/5 号房是"必备弹药"，2/6 号房是"高级弹药"

## 四、Claude Code 协作三原则

### 原则 1｜每次 session 开始时"重新加载上下文"
Claude Code 是 stateless 的（每次 `claude` 命令都是新会话）。所以每次开工先跑 Step 2 那段初始化 prompt——10 秒事，但能确保它"记得"是谁的助手。

### 原则 2｜让它"边做边汇报"
默认情况下让 Claude Code 在每个关键步骤汇报：
- "我准备调 X skill" 
- "我读到了 Y wiki 内容"
- "草稿写到一半，第 3 段我不太确定，要继续吗"

这样你能随时干预，不至于跑完才发现方向错了。

### 原则 3｜调用记录写进输出
**所有产出必须列出"调用了哪些 wiki 资源"**——这是公众号文章那篇的附录设计。

价值：
- 可追溯（出问题知道哪张信息卡的锅）
- 透明度（学员能看到 AI 的决策路径）
- 是真正"AI 调用知识库"的可视化证据

## 五、WorkBuddy 移植指南

> Claude Code 版验证成功后，把这套搬到 WorkBuddy 才能实现"24 小时自动跑"。

### 5.1 哪些可以直接复用

| 资产 | 直接复用 | 需要调整 |
|---|---|---|
| 6 个 SKILL.md | ✅ 直接拷 | — |
| prompts/ 子目录 | ✅ 直接拷 | — |
| references/ 子目录 | ✅ 直接拷 | — |
| Wiki 底座（六间房） | ✅ 直接读取 | 需配置 WorkBuddy 的"知识库绑定"指向 ob大脑路径 |
| CLAUDE.md | ✅ 直接读取 | — |
| 内容助理 agent 档案 | ⚠️ 需翻译 | WorkBuddy 用自己的 agent 配置格式，需要把 .md 转换 |

### 5.2 需要在 WorkBuddy 里新建的

1. **Automation 触发器**（让 skill 不需要每次手动 `claude` 启动）
   - 微信收到消息 → 自动触发 `wechat-ingest`
   - 每天 21:00 → 自动触发 `daily-brief`
   - 每周日 20:00 → 自动触发"下周选题生成"

2. **ClawBot 入口绑定**
   - 让微信消息能进入 WorkBuddy
   - 让 WorkBuddy 输出能推送回微信

3. **意图识别中间层**
   - WorkBuddy 收到微信消息后，先判断意图（入库/Query/创作？）
   - 然后路由到对应 skill
   - 这一层在 Claude Code 版是手动指令完成，在 WorkBuddy 版需自动化

### 5.3 移植步骤（高层）

```
Step 1：在 WorkBuddy 创建项目，绑定 ob大脑文件夹
Step 2：拷贝 05_System/skills/ 到 WorkBuddy 的 skill 注册表
Step 3：把 内容助理.md 翻译为 WorkBuddy agent 配置
Step 4：配 ClawBot + 微信
Step 5：配 4 个 Automation 触发器（碎片入库/日报/周报/月报）
Step 6：跑通端到端测试（微信发消息 → 入库 → 收到回复）
```

> 💡 **建议**：移植前先在 Claude Code 版用 1 周，验证哪些 skill 真的常用、哪些其实不需要——避免在 WorkBuddy 里搭一堆没人用的自动化。

## 七、内容助手 vs 单一 prompt 的本质区别

很多人一开始会问："为什么不直接用一个长 prompt？"

| 维度 | 长 prompt | 内容助手系统 |
|---|---|---|
| 复用性 | 每次复制粘贴 | 触发词调用 |
| 维护性 | 改一处要全部同步 | 改 SKILL.md 一处生效 |
| 调用透明度 | 黑盒 | 输出末尾列出调用资源 |
| 多场景适配 | 写一个超长 prompt | 6 个 skill 各司其职 |
| 团队/学员复用 | 只能给文本 | 给一套文件夹结构 |
| 自动化潜力 | 几乎无 | 能接 Automation |

**底层逻辑**：把"工作流"沉淀为"资产"，而不是每次都从零写 prompt。
